# forge-implement — Framework

Mental model for `forge-implement`: the orchestration skill that takes an audited Forge spec corpus and drives implementation by delegating to subagents. A prior `forge-armour` pass is recommended for security-exposed projects but not mandatory. This is **not** the SKILL.md; it is the source of truth from which the skill artifact is authored.

Audiences:
- An LLM / agent executing the orchestrator
- An engineer authoring `SKILL.md` from this framework
- A human wanting to understand what happens during a run

---

## 1. What `forge-implement` is

**Purpose.** Take a completed, audited Forge spec corpus and produce working implementation: code files, test files, scaffolding, and (optionally) git commits. Respects the spec-driven contract by never shortcutting the pipeline — tests precede implementation, subagents are isolated, failures are diagnosed cleanly.

**Scope contrast with the four prior skills:**

| | discover / decompose / forge-atom / forge-audit | **forge-implement** |
|---|---|---|
| Role | Spec creation / review | Spec → code translation |
| Writes | Spec files | Code files + test files |
| Work type | Interviewing / challenging | Orchestrating subagents |
| Delegates | No | Yes — spawns `forge-test-writer` and `forge-implementer` |
| Touches L0/L2 | Yes (discover, forge-atom) / no (decompose, audit) | **No** — specs are frozen |

**Outputs at exit:**

| Artifact | Contains |
|---|---|
| `<spec-dir>/implementation-plan.yaml` | The build plan: units + dep graph + architecture section + status |
| `implementation/progress.yaml` | Live execution state: per-unit status, attempt history, coverage numbers |
| Generated code and test files | At target paths declared in the plan |
| `implementation/attempts/<entity>/` | Stashed attempts, only present on final failures |
| `tests/<entity>.audit.md` | Per-entity test audit files (I5) |

---

## 2. Operating principles

1. **Orchestrator is a pure dispatcher.** Never makes implementation decisions. Never inlines spec content in subagent prompts. Subagents use `forge context` to load their own state.
2. **Test-before-implementation isolation.** Test-writer subagent runs first, completes (with red-phase verification), then implementation subagent runs in a fresh session, blind to tests.
3. **Architecture is declared once and respected throughout.** J1/J2/J3 — architecture section is assembled, human-confirmed, then locked. Subagents may bail with `architecture_conflict` but never mutate the architecture.
4. **Minimality in generated output.** Subagents produce the smallest code satisfying the spec. No speculative layers, helpers, or boilerplate.
5. **Specs are frozen during a run.** Never modify spec files. Changes require re-audit + re-run.
6. **Dep graph drives ordering.** Units run in topological order across modules; independent units parallelize per `--parallelism`.
7. **Retries are spec-linked, not test-linked.** Implementation retries receive sanitized feedback referencing spec elements — never test assertion diffs.
8. **Partial completion is fine.** Succeeded units stay; failed units stash attempts; blocked units wait. Resumable via `--resume`.

---

## 3. Pipeline overview

```
/forge-implement
  ↓
Step 1: Load or generate plan + architecture (§4)
Step 2: Audit gate check (§5)
Step 3: Generate shared scaffolding (§6)
Step 4: Execute the plan (§7)
Step 5: Final rollups (§8)
Step 6: Handover (§9)
```

## 4. Plan + architecture generation

### Plan file: `<spec-dir>/implementation-plan.yaml`

Shape (A5):

```yaml
# Generated by forge-implement
generated_at: <ISO-8601>
scope: project-wide

architecture:
  # per J2 — assembled from spec inference + UI consultation + human edits
  default_source_root: src/
  source_root_overrides: {}
  atom_naming_policy:
    typescript: camelCase_per_atom_file
    python: snake_case_per_atom_file
    go: snake_case_per_atom_file
  test_frameworks:
    typescript: vitest
    python: pytest
  schema_tools: {}
  iac_tools: {}
  ui_framework: react
  ui_styling: css_modules
  error_handling:
    typescript: exceptions
    go: error_return
    python: exceptions
  dependency_injection: direct_imports
  async_patterns:
    typescript: async_await

units:
  - id:            atm.pay.charges_schema
    kind:          atom_DECLARATIVE
    depends_on:    []
    status:        pending
    target_files:
      - src/pay/schema.prisma
      - tests/unit/pay/schema.test.ts

  - id:            atm.pay.persist_charge
    kind:          atom_PROCEDURAL
    depends_on:    [atm.pay.charges_schema]
    status:        pending
    target_files:
      - src/pay/persistCharge.ts
      - tests/unit/pay/persistCharge.test.ts

  # ... more units ...
```

### Drift detection (A3)

On invocation, if `implementation-plan.yaml` exists:
1. For each unit in the plan: does the referenced spec still exist? If deleted, mark unit as `orphaned`.
2. For each spec file mtime / hash: compare to the plan's `generated_at`. If any are newer: flag drift.
3. Summarize drift: *"Plan was generated on `<date>`. Since then: 2 new atoms added, 1 renamed, 3 modified. Regenerate plan, or continue with existing?"*

### Plan generation algorithm (B1-B4)

When generating from scratch or regenerating:

1. Run `forge list --kind atom --ids-only` + same for flow / journey / artifact — collect all entities.
2. For each entity:
   - Determine kind
   - Compute target_files via architecture's naming policy (E2)
   - Build depends_on:
     - For atom: atoms CALLed in logic + DECLARATIVE atoms that declare datastores referenced via `WRITES_DB` + artifacts in `READS_ARTIFACT` chain
     - For flow: atoms in sequence + atoms used as compensation
     - For journey: handler atoms + invoked flows
3. Topologically sort the dep graph. Break ties by module cohesion (C's variation of topological sort — prefer completing a module's atoms before switching modules).
4. Emit the `units` list.

### Architecture generation (J1, J2)

Before emitting the plan, assemble the architecture section:
1. **Inferred decisions** (A, B, C, D, F, G, H, I, J from locked design):
   - `default_source_root`: `src/`
   - `source_root_overrides`: for each module declaring `tech_stack.source_root`, include it
   - `atom_naming_policy`: per-language from E2's lookup
   - `test_frameworks`: per-module, infer from `tech_stack.frameworks` or fall back to language defaults
   - `schema_tools`: per-module from `tech_stack.schema_tool` or defaults
   - `iac_tools`: same
   - `error_handling`: language idioms (exceptions for JS/Py; error-return for Go; Result for Rust)
   - `dependency_injection`: default `direct_imports` unless frameworks list a DI framework (NestJS → `nest_di`)
   - `async_patterns`: language defaults

2. **UI consultation** (E — elicited, not inferred):
   - If any COMPONENT atoms exist in the project, pause and ask:
     - *"Project has N COMPONENT atoms. UI framework — inferred as `react` from `frameworks`. Confirm, or different?"*
     - *"Styling — inferred as `<best_match>` from frameworks. Alternatives: Tailwind / CSS modules / styled-components / vanilla CSS / other."*
     - *"Design system? (none / shadcn / Material / custom — describe)"*
     - *"Responsive strategy? (mobile-first / desktop-first / fluid)"*
     - *"Accessibility target? (WCAG 2.1 AA / not specified)"*
     - *"Theming? (light / dark / light-and-dark / multi-theme)"*
   - Record answers in `architecture.ui_*` fields.

3. Assemble the `architecture` block in the plan, then pause for human editing (A4).

### Human edit gate (A4)

After generating plan + architecture, pause with a summary:

```
Plan generated: 12 units in 3 modules.
Architecture generated: React + CSS modules, Prisma for PAY, Terraform for all infra.

Plan file: <spec-dir>/implementation-plan.yaml

Edit the plan now if you want (reorder units, modify architecture, adjust target files). When ready, confirm to proceed: y

```

Wait for confirmation. On confirmation, proceed to Step 2.

## 5. Audit gate (H1)

Before execution:

1. Check `<spec-dir>/audit-history.md` exists. If not: *"No audit history found. Run `/forge-audit` before implementation."*
2. Find the most recent audit timestamp.
3. Check every spec file's mtime vs the last audit. If any spec is newer than the last audit: *"Specs changed since last audit. Run `/forge-audit` (or `/forge-audit --tier quick`) before implementation."* (Can be overridden with `--skip-audit-check`.)
4. Parse `audit-history.md` for findings with `status: open` and severity `blocking`. If any: *"N blocking audit findings open. Confirm proceed [y/N] or `--force`."*
5. If clean: proceed silently.

---

## 6. Shared scaffolding generation (D6)

Before any subagents spawn:

1. **Types** — For each language in the architecture, generate a types module that materializes every L0 type as a language-native type:
   - TypeScript: `src/generated/types.ts` with `interface reg_pay_Charge { ... }` for each L0 entity type, `enum reg_usr_Status` for each enum.
   - Python: `src/generated/types.py` with pydantic / dataclass per L0 entity.
   - Go: `internal/generated/types.go` with structs.
   Imports: scoped by module (e.g., `src/generated/types/pay.ts`) or unified (e.g., `src/generated/types.ts`) — architecture decides.

2. **Constants** — For each language: `src/generated/constants.ts` (etc.) exporting every L0 constant as a language-native constant.

3. **Fixture factories** — For each module, generate `tests/fixtures/<mod>.fixtures.ts` (etc.) with a factory per entity type that produces valid test data with sane defaults + overrides. Factories use architecture's type scaffolding.

These files are regenerated every run — they are purely spec-derived. Orchestrator writes them; no subagents involved.

## 7. Execution (§7)

### Main loop

```
while unfinished units exist:
    ready = units where status == pending AND all depends_on are done
    if len(ready) == 0 and no in_flight:
        break  # all remaining are blocked; cannot progress
    
    spawn_slots = parallelism - len(in_flight)
    for unit in ready[:spawn_slots]:
        spawn_pipeline(unit)
    
    wait for any in_flight subagent to complete
    process result
    checkpoint progress
```

### Per-unit pipeline

For each unit (atom / flow / journey), the pipeline is:

```
1. Spawn test-writer subagent (forge-test-writer)
   - entity_id, level (unit/integration/system from kind), target_test_file, target_audit_file, architecture
   - Subagent runs forge context, generates tests, writes files
   - Returns summary

2. Red-phase verification (D3)
   - Run the test command against the (still-empty) target_source_file location
   - Expect failure (compile error or test failure — either counts)
   - If any test passes trivially: re-spawn test-writer with feedback "test X passed against empty stub"
   - Max N test-writer retries (default 2)
   - If test-writer exhausts retries: mark unit failed, continue with rest

3. Spawn implementation-writer subagent (forge-implementer)
   - entity_id, kind, target_source_file, architecture
   - Subagent loads spec via forge context, generates code, writes files
   - Returns summary (including architecture_conflict flag)
   - If architecture_conflict: mark unit failed, cancel all dependents, continue

4. Green-phase verification (F1)
   - Run tests against new implementation
   - If pass: mark unit done, proceed
   - If fail: re-spawn implementer with sanitized failure + spec-linked hints (D4)
     - Max N implementer retries (default 2)
     - If exhausted: mark unit failed, route-to-human diagnostic (D5), block dependents

5. Build check (F2)
   - For typed languages: run compile / type-check on just-written files
   - If fail: treat as test failure, retry

6. Structural sanity check (F5)
   - Verify types referenced, errors returned, markers backed by code patterns
   - Report as a low-severity warning if issues; don't fail

7. Format (F3)
   - Run architecture's formatter on written files

8. Update progress, checkpoint
```

### Parallelism coordination

- `--parallelism N` cap (default 4) enforces concurrent in-flight subagents.
- When a subagent completes (success or failure), orchestrator updates progress, re-checks the dep graph, and spawns any newly-ready units up to the cap.
- The retry policy (C5) applies per-unit — a unit in retry counts against parallelism until its final state.

### Status lifecycle per unit

```
pending → (when deps done and capacity available) → in_flight → done | failed | architecture_conflict
blocked (when a dep is failed or architecture_conflict) → remains blocked
```

Failed or blocked units do not prevent unrelated units from running.

---

## 8. Final rollups (F1, F2)

After the main loop:

1. **Full project test suite.** Run the entire project's test command (e.g., `vitest run`, `pytest`, `go test ./...`). Captures cross-atom interaction issues. Report pass/fail + summary.
2. **Full project build** (typed languages). Run `tsc --noEmit` / `go build ./...` / etc. Capture any type errors not caught per-file.
3. **Coverage report** (I4). Run the coverage tool and include numbers in the progress report.

Any failures at this stage don't retroactively fail the per-unit passes — they're reported as project-level findings.

## 9. Handover (H3)

Assemble the summary:

```
forge-implement complete.
Scope:           project-wide
Units done:      N
Units failed:    M
Units blocked:   K

Coverage (after rollup):
  line:   <pct>%
  branch: <pct>%

Full project test suite: <pass|fail>
Full project build:      <pass|fail>

Failed units (inspect stash):
  - <id> — <reason>  [stash: implementation/attempts/<id>/]
  - ...

Commit these changes as per-atom commits? [y/N]
```

If human accepts: for each successful unit, run `git add <target_files> && git commit -m "feat(<mod>): implement <entity_id>"`. If any atoms were part of a larger group (e.g., DECLARATIVE atom + its consumers committed together), use a single commit per related group.

If human declines: leave all files uncommitted.

Recommend next steps:
- If failures: *"Inspect `implementation/attempts/` stashes. Re-run `/forge-audit --scope <failed_id>` for spec review, or `/forge-atom <failed_id>` to revise the spec, then `/forge-implement --resume`."*
- If clean: *"Implementation complete. Repo ready. Run your test suite / CI / deploy pipeline."*

## 10. Failure, retry, resumability

### Retry protocol (C5, D4, D5)

- **Test-writer retries:** up to N=2. Feedback: specific trivially-passing test names + instruction to make them check real behavior.
- **Implementer retries:** up to N=2. Feedback: sanitized test output with assertion diffs stripped + spec element each failing test derives from.
- **After exhaustion:** mark unit failed; stash all attempts in `implementation/attempts/<entity>/attempt-<N>/`; block direct dependents; continue unrelated branches.

### Resumability (G3)

On `/forge-implement` invocation when progress.yaml exists:

1. Read progress.yaml. Show summary: *"Last run: N done, M failed, K pending, J in_flight, L blocked. Last activity: `<timestamp>`."*
2. Validate stashed state:
   - Every `status: done` unit's target files still exist?
   - Have any spec files been modified since progress.generated_at?
3. Report validation:
   - Clean: *"Resume, restart, or review? [r/s/v]"*
   - Drift detected: *"3 specs have changed since last run. Resume risks inconsistency. Options: (a) resume anyway; (b) regenerate plan; (c) show what changed."*
4. Human decides.

### Resume flags (G5)

- `--resume` — continue from progress
- `--skip <entity_id>` — mark unit as manually handled; skip in this run
- `--manual <entity_id>` — same as skip, but tracked in audit trail
- `--retry-override <entity_id>=<N>` — increase retry limit for specific unit
- `--restart <entity_id>` — wipe unit's progress + stashes, re-run from scratch

## 11. What `forge-implement` does NOT do

- **Does not create or edit specs.** Ever. Frozen from run start.
- **Does not make implementation decisions.** All decisions in architecture or deferred to subagent per their spec.
- **Does not run interactive interviews.** Unlike discover / forge-atom, this skill is execution-oriented. Only interactive moments: A4 plan-edit pause, J2 UI consultation, G3 resume confirmation, H3 commit-offer.
- **Does not generate flows / journeys / artifacts from nothing.** These must exist as specs already.
- **Does not train MODEL atoms.** Produces stubs delegating to fallback; training is out of scope.
- **Does not run migrations / deploy infrastructure.** DECLARATIVE atoms produce migration files or IaC files; running them is a human / CI concern.

## 12. Artifact schemas

### `implementation-plan.yaml`

See the full shape in §4 above. Key fields:
- `generated_at`, `scope`
- `architecture` block (all J2 decisions)
- `units` list (each with `id`, `kind`, `depends_on`, `status`, `target_files`)

### `implementation/progress.yaml`

Live execution state, checkpointed after every subagent completion:

```yaml
plan_ref: <spec-dir>/implementation-plan.yaml
plan_generated_at: <ISO-8601>
progress_updated_at: <ISO-8601>

units:
  - id: atm.pay.persist_charge
    status: done
    attempts:
      test_writer: 1
      implementer: 1
    tests_passed: 14
    tests_failed: 0
    coverage:
      line_pct: 94
      branch_pct: 88
    completed_at: <ISO-8601>

  - id: atm.pay.charge_card
    status: failed
    attempts:
      test_writer: 1
      implementer: 2
    failure_reason: "Implementation failed 2 retries on spec case 'amount <= 0 rejected with PAY.VAL.001'"
    stashed_attempts: implementation/attempts/atm.pay.charge_card/
    blocking_deps: []
```

### `implementation/attempts/<entity>/attempt-<N>/`

Stashed per G4 only on final failure. Contains: the last-attempt source file, the test output, the retry feedback sent to the subagent.

## 14. Open design questions (for future iteration)

- **Parallel test-writer + implementer per unit.** Currently sequential per unit (test-writer completes, then implementer). Could run concurrently if both derive purely from spec. Would need careful handling of the red-phase verification. Defer.
- **Per-atom architecture overrides.** J3 locked no mid-run mutations; atoms can't override architecture. If real projects demand per-atom exceptions, add `spec.architecture_override` block. Defer until use case emerges.
- **Multi-language projects per module.** Current model assumes one language per module. If a module genuinely has multi-language atoms (e.g., a Python backend atom + a TypeScript client atom), architecture needs per-atom language. Not supported now.
- **Contract tests as separate subagent type.** Currently part of forge-test-writer's scope. If contract tests become complex, split into `forge-contract-test-writer`.
- **Integration tests for cross-module flows.** Currently forge-test-writer handles unit/integration/system based on level parameter. Integration tests across module boundaries may benefit from a dedicated subagent type. Defer.
